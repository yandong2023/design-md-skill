# README and markdown page structures

Use these structures as starting points. Adapt them to the product rather than copying mechanically.

## 1. Open-source devtool README

1. Project name
2. One-line value proposition
3. Product snapshot or screenshot
4. Why this exists
5. Key features
6. Quick start
7. Example workflows
8. How it works
9. Limitations / trade-offs
10. Contributing

## 2. AI tool README

1. Project name
2. Clear claim with scope
3. Trust qualifier or proof line
4. Screenshot / CLI demo / sample output
5. What problem it solves
6. Core workflow
7. Quick start
8. Common use cases
9. Architecture or implementation notes
10. Limitations and future work

## 3. Docs homepage

1. One-line orientation
2. Who this is for
3. Start here
4. Core concepts
5. Common tasks
6. Advanced topics
7. Reference/API links

## 4. Product landing page

1. Hero
2. Problem framing
3. Solution explanation
4. Proof
5. Use cases
6. Differentiation
7. FAQ
8. CTA

## Structure checks

Before keeping a structure, verify:
- the first screen explains what the product is
- the audience is obvious
- there is a next action
- proof appears before trust is exhausted
- technical detail is not blocking initial understanding
