# Codex usage patterns

Use these patterns when applying the skill in a real coding or editing session.

## Default operating order

1. Identify the target file or page
2. Classify the page type
3. Diagnose the top 3–5 problems
4. Decide whether to critique, plan, or edit directly
5. Rewrite the most leveraged sections first
6. Re-check for trust, clarity, and CTA strength

## Preferred deliverable modes

### 1. Critique mode
Use when the user wants analysis before changes.

Return:
- summary judgment
- top issues
- why they matter
- recommended fix order

### 2. Rewrite-plan mode
Use when the user wants structure before file edits.

Return:
- proposed new structure
- new hero block
- replacement section titles
- CTA recommendation
- proof placement recommendation

### 3. Direct-edit mode
Use when the user wants implementation.

Do:
- edit the file directly
- preserve all true facts
- avoid inflating unsupported claims
- keep installation and docs paths easy to find

After editing, summarize:
- what changed
- why it is better
- any missing proof that still limits the page

## High-value targets for early rewrites

Rewrite these first when time is limited:
- H1
- subheadline
- top CTA
- first paragraph after hero
- first section title
- quickstart intro

## Codex-specific behavior

When editing files, prefer:
- minimal, high-leverage changes first
- preserving existing code structure when possible
- changing copy before changing layout markup
- restructuring sections only when current order is clearly hurting clarity

## Do not do this

- do not invent metrics, traction, customers, or benchmarks
- do not make every page sound like a YC landing page
- do not bury install instructions for a devtool
- do not remove useful technical specificity just to sound smoother
