# Headline patterns

Use these as patterns, not templates to copy blindly.

## Good headline traits

A strong headline is usually:
- short
- specific
- outcome-aware
- easy to repeat

## Useful headline formulas

### 1. What it is + why it matters
- The local-first memory layer for AI workflows
- A faster way to review developer docs before shipping

### 2. Outcome-first
- Turn scattered product docs into a clearer developer onboarding path
- Make your README feel like a real product page

### 3. Replacement framing
- Replace bloated setup guides with a README developers can actually use
- Stop shipping homepage copy that reads like internal notes

### 4. Constraint-aware trust framing
- Developer-friendly product pages without fake marketing fluff
- Clearer AI product messaging, with proof and without hype

## Subheadline patterns

A good subheadline should clarify:
- who it is for
- what it changes
- how it works at a high level

Examples:
- Improve README structure, hero messaging, and CTA flow for open-source and devtool products.
- Critique and rewrite markdown-first product presentation for GitHub-first launches, docs homepages, and landing pages.

## Proof line patterns

Use a short proof line under the hero when possible:
- Open source. Local-first. No cloud dependency.
- Used for README, docs, and landing-page rewrites.
- Designed for developer-facing products and AI tools.

## Warnings

Avoid headlines that are:
- generic (“The future of X”)
- overloaded with buzzwords
- technically precise but emotionally dead
- emotionally loud without evidence
